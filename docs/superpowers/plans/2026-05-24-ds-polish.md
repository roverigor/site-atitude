# Design System Polish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Resolve the 51 🟡 YELLOW findings deferred during the DS rollout (PR #2) — typography utility → DS class swaps, raw duration values → DS motion tokens, `bg-white` → semantic background tokens, eyebrow class adoption, and a handful of small motion/spacing polish items across 11 audit docs.

**Architecture:** Same audit-first model as the rollout. Source-of-truth for each polish task is the existing audit doc in `docs/ds-audit/NN-name.md` — specifically the items marked 🟡 YELLOW with a `Fix:` note. No new auditing. Per piece: read YELLOWs from the audit, apply each as a separate commit, update the audit doc to mark the YELLOW as resolved, then move to the next piece. Single feature branch `chore/ds-polish` from `main`, single PR back to `main`.

**Tech Stack:** Same as rollout — Next.js 15.5 (App Router), React 19, Tailwind 4 `@theme` CSS-first, framer-motion 12, Poppins self-hosted. Token source: `src/app/globals.css`. Vivo references: `src/components/home/*`, `src/app/cursos/**`, `src/components/ui/*`.

**Spec (implicit):** The YELLOW items in `docs/ds-audit/00-11-*.md`. No separate spec doc — the audit findings are the requirements.

- Polish fix: `polish(<scope>): <one-line>` — distinct prefix from rollout's `redesign(...)` / `fix(...)` so the polish PR is easy to identify in git log
- Audit doc update: `chore(ds-audit): mark <piece> YELLOWs resolved`
- Branch scaffold/index: `chore(ds-polish): scaffold` and `chore(ds-polish): mark <piece> done in polish index`

**Per-fix workflow (used inside every Task N's Step 4):**

For each numbered YELLOW item in the Fix Plan (derived from the audit doc):
- **a.** Read the YELLOW line in the audit doc — note its `Fix:` recipe
- **b.** Make the smallest possible edit to the source file(s) that resolves only that finding
- **c.** Run V-LINT and V-BUILD — both must pass before commit
- **d.** Commit:
  ```bash
  git add <file>
  git commit -m "$(cat <<'EOF'
polish(<scope>): <one-line>

Resolves YELLOW from docs/ds-audit/<audit-doc>.md

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
  ```
- **e.** In the audit doc, change the YELLOW line's emoji from 🟡 to ✅ and append ` — Resolved in <commit-sha>` after the existing text. Do this for every resolved item before the next Step.

**YELLOW items that MUST stay deferred (do NOT fix in this PR):**

Each piece's audit doc may flag YELLOW items that depend on:
- Cross-component refactors (e.g. MobileNav focus restoration needs Header trigger ref)
- New DS token definitions (e.g. extending Badge to accept opacity-safe tokens)
- framer-motion structural changes (e.g. honoring `prefers-reduced-motion` globally)

If a YELLOW says "structural change" or "extend DS" in its `Fix:` note, leave it 🟡 in the audit doc and explicitly call it out in the task's report as `still-deferred`. Do not invent fixes for these.

**Token guardrail:** Verify every cited DS token by `grep -nE "^\s*--<token-name>\b" src/app/globals.css` BEFORE using. Same rule as rollout — no invented tokens.
